# Message House Architecture

## Structure

```
                 [PRIMARY MESSAGE]
                       (one sentence — wedge in plain language)
                              |
        --------------------------------------------
        |                     |                    |
  [PILLAR 1]            [PILLAR 2]           [PILLAR 3]
  (sub-message that proves primary; maps to top buyer values)
       |                     |                    |
   [PROOF]              [PROOF]             [PROOF]
   (case study / data / customer quote / demoable feature)
```

## Why Pillars Beat Features

Messages stack. A buyer remembers 1 primary + 3 sub-messages. They don't remember 12 features.

## Rules

- Primary: 1 sentence, the wedge in plain language
- 3 pillars, each mapping to a top buyer value
- ≥2 proofs per pillar (case study, data, customer quote, demoable feature)
- If you can't prove a pillar, flag it for testing — don't leave it unverified

## Worked Example — WorkflowDoc (fictional)

**Primary:** "Turn your closed tickets into a self-updating support runbook library — in days, not weeks."

**Pillar 1:** "AI-native, not bolt-on" — Proofs: 8x faster authoring vs Guru; auto-detects stale runbooks; drafts from tickets not prompts

**Pillar 2:** "Built for support, not for everyone" — Proofs: $400/seat vs $25k+ Guru ACV; built around support workflow; champion = Support Ops Manager

**Pillar 3:** "Live in days, not weeks" — Proofs: First runbook in Day 2 (4/4 pilots); native Zendesk + Intercom; 3.5 wk sales cycle avg
